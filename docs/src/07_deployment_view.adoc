== Deployment View

=== Infrastructure Level 1

El sistema se despliega en una instancia **AWS EC2** (Ubuntu 24.04) con una
**Elastic IP** fija, usando **Docker Compose** para orquestar todos los
contenedores. Las imágenes se publican automáticamente en
**GitHub Container Registry (ghcr.io)** mediante GitHub Actions al crear una
release.

[plantuml, deployment-level1, png]
----
@startuml
node "AWS EC2 (Ubuntu 24.04)" {
    node "Docker Compose" {
        component "webapp\n(nginx:80)" as webapp
        component "users\n(Node.js:3000)" as users
        component "gamey\n(Rust/Axum:4000)" as gamey
        component "mongodb\n(mongo:27017)" as mongodb
        component "prometheus\n(:9090)" as prometheus
        component "grafana\n(:9091)" as grafana
    }
}

cloud "MongoDB Atlas" as atlas
cloud "GitHub Container Registry\n(ghcr.io)" as ghcr

webapp --> users : HTTP :3000
webapp --> gamey : HTTP :4000
users --> atlas : TLS :27017
gamey --> atlas : TLS :27017
ghcr --> webapp : docker pull
ghcr --> users : docker pull
ghcr --> gamey : docker pull
@enduml
----

Motivation::
Se eligió AWS EC2 porque proporciona una máquina virtual con IP pública
configurable. La Elastic IP garantiza que la dirección no cambia al reiniciar
la instancia, lo que evita tener que actualizar los secrets de GitHub Actions
y reconstruir las imágenes cada vez. Docker Compose simplifica el despliegue
al gestionar todos los servicios como una unidad.

Quality and/or Performance Features::
* **Disponibilidad**: los contenedores se configuran con `restart: always`
para que arranquen automáticamente si la instancia se reinicia.
* **Seguridad**: las contraseñas y URIs sensibles se inyectan como variables
de entorno en tiempo de ejecución, nunca se incluyen en las imágenes.
* **Escalabilidad**: la arquitectura de microservicios permite escalar cada
servicio de forma independiente si fuera necesario.
* **Observabilidad**: Prometheus recoge métricas del servicio `users` y
Grafana las visualiza en dashboards.

=== Infrastructure Level 2

==== Pipeline CI/CD (GitHub Actions)

El despliegue se activa automáticamente al publicar una **release** en GitHub.
El workflow `.github/workflows/release-deploy.yml` ejecuta las siguientes fases
en orden:

[plantuml, deployment-cicd, png]
----
@startuml
|GitHub Actions|
start
:Tests unitarios\n(Node.js + Rust);
:Tests E2E\n(Cucumber + Playwright);
fork
  :Build y push\nwebapp → ghcr.io;
fork again
  :Build y push\nusers → ghcr.io;
fork again
  :Build y push\ngamey → ghcr.io;
end fork
:Deploy SSH → AWS EC2\nwget docker-compose.yml\ndocker compose up -d --pull always;
stop
@enduml
----

[cols="1,3"]
|===
|Paso |Detalle

|Tests unitarios
|`cargo test` para gamey, `npm test` para webapp y users en paralelo.

|Tests E2E
|`cucumber-js` con Playwright arrancando `users` y `webapp` concurrentemente.
Se ejecuta con `continue-on-error: true` para no bloquear el despliegue.

|Build imágenes
|El Dockerfile de webapp recibe `VITE_API_URL` y `VITE_GAMEY_URL` como
`ARG` en tiempo de build para incrustar la IP pública de la instancia.

|Deploy SSH
|GitHub Actions conecta por SSH a la instancia EC2 usando la clave privada
almacenada en el secret `DEPLOY_KEY`. Descarga el `docker-compose.yml` desde
el repositorio y levanta los contenedores con `--pull always` para usar
siempre las imágenes recién publicadas.
|===

==== MongoDB Atlas

La base de datos de producción es un cluster de **MongoDB Atlas** con
replicación en tres shards. La conexión usa TLS y autenticación por usuario
y contraseña.

[cols="1,3"]
|===
|Parámetro |Valor

|Host
|`ac-mybgnhh-shard-00-0{0,1,2}.yt6wilm.mongodb.net:27017`

|Base de datos (users)
|`yovi2c_db`, colección `users`

|Base de datos (gamey)
|`yovi2c_db`, colecciones `games` y `sessions`

|Autenticación
|Usuario `yovi2c_db_user`, contraseña inyectada via variable de entorno

|TLS
|Activado (`tls=true`)
|===

==== Entornos

[cols="1,2,2"]
|===
|Entorno |Base de datos |URLs de la API

|Desarrollo local
|MongoDB local en `mongodb://mongodb:27017` (Docker Compose)
o `mongodb://localhost:27017` (sin Docker)
|`http://localhost:3000` y `http://localhost:4000`
definidas en `webapp/.env.local`

|Producción (AWS EC2)
|MongoDB Atlas con URI completa inyectada via `DB_URI`
|`http://<ELASTIC_IP>:3000` y `http://<ELASTIC_IP>:4000`
incrustadas en el build de la imagen Docker
|===
